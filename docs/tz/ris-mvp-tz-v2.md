## 0. Контекст учебного проекта

Данное ТЗ является основой для учебного проекта по разработке медицинского программного
обеспечения. Команда получает реальный продуктовый контекст — систему, которая используется
в диагностических центрах.

**Что строим:** MVP (Minimum Viable Product) радиологической информационной системы (RIS)
для рынка Казахстана.

**Чему учимся:**
- Работа с требованиями к медицинскому ПО
- Интеграция с внешними протоколами (DICOM)
- Проектирование RBAC и аудита
- Жизненный цикл медицинских данных

**Успешный результат MVP** — работающая система, в которой регистратор может создать
пациента и направление, лаборант — зафиксировать получение снимков, а врач-рентгенолог —
оформить и подписать заключение.

## 1. Назначение и контекст

Радиологическая информационная система (RIS) автоматизирует рабочий процесс отделения
лучевой диагностики: от регистрации пациента и направления — до проведения исследования,
описания врачом-рентгенологом и выдачи заключения.

**Классический поток:**
```
Регистрация пациента
      ↓
Создание направления (услуга, приоритет, МКБ-10)
      ↓
Запись в расписание (аппарат, слот времени)
      ↓
Передача задания на аппарат через DICOM MWL
      ↓
Проведение исследования → приём снимков (C-STORE → Orthanc)
      ↓
Контроль качества снимков (лаборант)
      ↓
Описание по шаблону протокола (врач-рентгенолог)
      ↓
Подпись и выдача заключения
      ↓
Статистика и метрики (turnaround time)
```

**Контекст MVP:** самостоятельная (standalone) система для частного диагностического
центра. Архитектура **integration-ready** — приём направлений изолирован за адаптером,
чтобы позже добавить интеграцию с МИС (HL7) без переписывания ядра.

### Ориентиры (best practices ведущих систем)
- **GE Centricity RIS** — веб-описание, структурированные шаблоны протоколов, метрики
  turnaround time, дашборды нагрузки.
- **Agfa Enterprise Imaging** — задача-ориентированный workflow: исследование как набор
  задач (контроль качества → описание → подпись).
- **Epic Radiant, Siemens RIS** — единая карточка пациента, клинические worklist'ы.
- **Orthanc / dcm4chee / ThaiRIS** — зрелые open-source DICOM-компоненты.

## 2. Границы MVP

### В составе MVP
- Регистратура: поиск, создание и редактирование карточки пациента.
- Направления: создание, управление статусами, печать.
- Расписание: календарь по аппаратам, запись, отметка явки.
- DICOM Modality Worklist (MWL), приём снимков (C-STORE), просмотр снимков через Orthanc.
- Описание исследований по шаблонам, подпись (ЭЦП-ready), выдача заключений в PDF.
- 6 ролей с ролевым доступом (RBAC).
- Справочники РК: тарификатор услуг, МКБ-10, аппараты, шаблоны протоколов.
- Аудит всех действий с медданными.
- Статистика и метрики (turnaround time, нагрузка).
- cito-оповещения направителю при критической находке.
- Второе мнение / рецензия (инициирует заведующий).

### Вне MVP (отложено, но с архитектурным заделом)
- Личный кабинет пациента.
- Полноценный биллинг (тарификатор используется как справочник услуг с тарифом).
- Интеграция с МИС по HL7 (только контракт адаптера).
- Интеграция с нацсистемами РК; СМС-оповещения пациенту.
- Казахская локализация контента (i18n-задел в архитектуре есть, перевод позже).
- Квалифицированная ЭЦП через NCALayer / НУЦ РК (логика подписи готова, интеграция позже).

## 3. Архитектура

**Вариант A — модульный монолит + Orthanc как отдельный DICOM-узел.**

```
┌─────────────────────────────────────────────────────────────┐
│              Веб-фронтенд (SPA)                             │
│        Рабочие места 6 ролей (React / Vue)                  │
└────────────────────────┬────────────────────────────────────┘
                         │  REST/JSON + WebSocket
┌────────────────────────▼────────────────────────────────────┐
│              RIS Backend (модульный монолит)                 │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────────┐   │
│  │Пациенты/ │ │Направл./ │ │Описания/ │ │Пользователи/ │   │
│  │регистр.  │ │расписание│ │подпись   │ │роли / аудит  │   │
│  └──────────┘ └──────────┘ └──────────┘ └──────────────┘   │
│  ┌──────────┐ ┌──────────────────────────────────────────┐  │
│  │Справочн. │ │Статистика / Метрики / Интеграц. адаптер  │  │
│  └──────────┘ └──────────────────────────────────────────┘  │
└──────┬──────────────────────────────────────┬───────────────┘
       │  SQL                                 │  DICOMweb / REST
┌──────▼──────┐                    ┌──────────▼──────────────┐
│ PostgreSQL  │                    │        Orthanc           │
│   (RIS DB)  │                    │  MWL · C-STORE · вьюер  │
└─────────────┘                    │  Stone Web Viewer        │
                                   └──────────┬──────────────┘
                                              │  DICOM
                                   ┌──────────▼──────────────┐
                                   │  Модальности            │
                                   │  КТ, МРТ, РГ, УЗИ, ММГ │
                                   └─────────────────────────┘
```

### Принципы архитектуры

| Принцип | Описание |
|---|---|
| Разделение ответственности | RIS ведёт рабочий процесс и данные; Orthanc хранит и показывает DICOM. RIS не дублирует пиксельные данные, ссылается по идентификаторам |
| Связка по Accession Number | Accession Number протягивается через весь цикл: заказ → MWL → снимки → автосопоставление |
| Интеграционный адаптер | Изолирует внешний мир (сейчас Orthanc; задел — HL7/МИС, нацсистемы РК) |
| Сквозной аудит | Все действия с медданными фиксируются в неизменяемом лог-журнале |
| Чёткие границы модулей | Каждый модуль тестируется независимо; задел под выделение в сервисы при росте |

**Деплой-топология:** RIS backend, БД и Orthanc — отдельные контейнеры (Docker Compose).
Orthanc хранит DICOM на своём томе, RIS — свою SQL-БД; раздельные бэкапы.

## 4. Роли и рабочие места

| Роль | Основные функции |
|---|---|
| **Регистратор** | Поиск/создание пациента; создание направления; запись в расписание; отметка явки; печать талона/направления; рабочий список записей |
| **Рентгенолаборант** | Рабочий список аппарата; статусы «прибыл / начато / снимки получены»; контроль качества (принять / переснять) |
| **Врач-рентгенолог** | Рабочий список «к описанию»; просмотр снимков (вьюер Orthanc); описание по шаблону + МКБ-10; подпись и выдача заключения |
| **Заведующий отделением** | Обзор нагрузки и статусов; переназначение описаний; инициация второго мнения/рецензии; контроль просроченных (turnaround time) |
| **Врач-направитель** | Создание направлений своим пациентам; просмотр статуса и готовых заключений (только чтение, только свои пациенты) |
| **Администратор** | Пользователи и роли; справочники (услуги, аппараты, шаблоны, МКБ); аудит, статистика, системные настройки |

**Доступ:** RBAC, принцип наименьших привилегий. Врач-направитель видит только своих
пациентов. Доступ к снимкам — только через RIS-сессию (вьюер Orthanc не публичен).

## 5. Функциональные требования

### M1. Пациенты и регистратура

| ID | Требование | Критерии приёмки |
|---|---|---|
| F1.1 | Поиск пациента по ИИН / ФИО / дате рождения / телефону | Результат появляется при вводе ≥ 3 символов; неточный поиск по ФИО |
| F1.2 | Создание/редактирование карточки: ИИН, ФИО, пол, дата рождения, контакты, льготная категория | ИИН валидируется (12 цифр, контрольная цифра); поля обязательности по НПА |
| F1.3 | Контроль дублей по ИИН | При совпадении ИИН — предупреждение с ссылкой на существующего пациента |
| F1.4 | История исследований пациента | Хронологический список направлений и заключений с фильтром по периоду |

### M2. Направления (orders)

| ID | Требование | Критерии приёмки |
|---|---|---|
| F2.1 | Создание направления: пациент → услуга (тарификатор) → модальность → область → врач-направитель → приоритет (плановый / срочный / cito) → предв. диагноз (МКБ-10) → клинические сведения | Все обязательные поля заполнены; направление сохраняется в статусе NEW |
| F2.2 | Генерация Accession Number и идентификатора заказа | AN уникален, формат согласован с командой (например, ГГ-ММДД-NNNNN) |
| F2.3 | Управление статусами направления (см. раздел 7) | Переходы только по разрешённым дугам; несанкционированные — ошибка 403 |
| F2.4 | Отмена направления с указанием причины | Отмена доступна только до начала исследования; причина обязательна |
| F2.5 | Печать направления (форма с реквизитами по НПА РК) | PDF содержит: организацию, врача, пациента, ИИН, услугу, код МКБ, дату |

### M3. Расписание и запись

| ID | Требование | Критерии приёмки |
|---|---|---|
| F3.1 | Календарь по аппаратам (день/неделя), слоты по длительности услуги | Слоты формируются автоматически исходя из рабочего времени аппарата и длительности услуги |
| F3.2 | Запись направления в свободный слот; перенос/отмена | Занятый слот недоступен; при переносе старый слот освобождается |
| F3.3 | Учёт рабочего времени аппарата и недоступности (ТО, поломка) | Администратор устанавливает окна недоступности; слоты в этот период не предлагаются |
| F3.4 | Отметка явки пациента («прибыл») | Статус направления переходит SCHEDULED → ARRIVED |
| F3.5 | Предупреждение о конфликтах / двойной записи | Система не позволяет записать один аппарат дважды в один слот |

### M4. Worklist и проведение исследования

| ID | Требование | Критерии приёмки |
|---|---|---|
| F4.1 | Публикация в DICOM MWL: демография + Accession Number + Study Instance UID + AE Title | Аппарат получает worklist-запись через C-FIND; данные совпадают с направлением |
| F4.2 | Рабочий список лаборанта по аппарату | Отображаются только исследования текущего аппарата; сортировка по времени записи |
| F4.3 | Статусы: к выполнению → прибыл → начато → снимки получены | Переходы фиксируются с timestamp; доступны только лаборанту своего аппарата |
| F4.4 | Автосопоставление студии с заказом по Accession Number / Study Instance UID | При совпадении — статус переходит в ACQUIRED автоматически за < 5 секунд |
| F4.5 | Ручное связывание студии и заказа при расхождении | Несопоставленные студии попадают в очередь; администратор/лаборант может связать вручную |
| F4.6 | Контроль качества: принять / переснять; комментарий лаборанта | «Переснять» возвращает статус в IN_PROGRESS; попытка сохраняется в истории |

### M5. Описание и заключение

| ID | Требование | Критерии приёмки |
|---|---|---|
| F5.1 | Рабочий список «к описанию» с фильтрами (модальность, приоритет, срок) | Сортировка: cito наверху, затем по времени; фильтры применяются без перезагрузки страницы |
| F5.2 | Экран описания: данные направления + встроенный вьюер Orthanc | Вьюер открывается по study_instance_uid; виден без перехода на отдельный URL |
| F5.3 | Шаблоны протоколов по модальности/области: структурированные поля + свободный текст | Шаблон подгружается автоматически по модальности; поля редактируемы |
| F5.4 | Заключение + код(ы) МКБ-10 | МКБ выбирается из справочника (поиск по коду и названию); минимум один код обязателен |
| F5.5 | Черновик → подпись (ЭЦП-ready: врач + дата/время + хэш содержимого) | После подписи документ недоступен для редактирования; хэш верифицируем |
| F5.6 | Выдача заключения (PDF); при critical_finding — cito-оповещение направителю | PDF содержит реквизиты по НПА РК; уведомление отправляется в системе |
| F5.7 | Второе мнение / рецензия (инициирует заведующий) | Создаётся отдельный Report, связанный с оригиналом; оригинал не перезаписывается |

### M6. Справочники

| ID | Требование | Критерии приёмки |
|---|---|---|
| F6.1 | Тарификатор услуг (код, наименование, тариф, модальность, длительность) | Импорт из Excel/CSV; версионирование (изменения не затрагивают старые записи) |
| F6.2 | МКБ-10 (поиск по коду/наименованию) | Полный справочник; поиск работает по частичному совпадению |
| F6.3 | Аппараты/модальности (тип, AE Title, расписание, статус) | AE Title уникален; изменение статуса аппарата блокирует запись на него |
| F6.4 | Шаблоны протоколов: создание/редактирование администратором | Шаблон привязан к услуге/модальности; механизм версионирования |
| F6.5 | Пользователи, роли, права | Администратор управляет учётными записями; смена пароля, блокировка |

### M7. Пользователи, доступ, аудит

| ID | Требование | Критерии приёмки |
|---|---|---|
| F7.1 | Аутентификация (логин/пароль; задел под 2FA) | Пароль хранится в виде хэша (bcrypt/argon2); сессия завершается после таймаута |
| F7.2 | RBAC: 6 ролей, разграничение доступа к данным | Попытка получить ресурс без прав возвращает 403; врач-направитель видит только своих пациентов |
| F7.3 | Сквозной аудит-лог: кто, что, когда (создание / изменение / просмотр / подпись) | Лог неизменяем; содержит: entity, action, user_id, timestamp, before/after |
| F7.4 | Неизменяемость подписанных документов; версионирование описаний | Попытка редактировать подписанный документ отклоняется API; правка создаёт новую версию |

### M8. Статистика и метрики

| ID | Требование | Критерии приёмки |
|---|---|---|
| F8.1 | Дашборд: число исследований по периодам / модальностям / врачам / аппаратам | Данные актуальны; фильтры по дате и модальности работают |
| F8.2 | Turnaround time (ISSUED − NEW) и доля просроченных | Метрика вычисляется корректно; настраиваемый порог «просрочено» |
| F8.3 | Нагрузка на аппараты и врачей | Отображается число исследований за период по каждому ресурсу |
| F8.4 | Выгрузка отчётов (CSV/Excel) | Файл содержит все строки выборки; кодировка UTF-8 |

### M9. Интеграционный адаптер

| ID | Требование | Критерии приёмки |
|---|---|---|
| F9.1 | DICOMweb/REST-обмен с Orthanc (MWL, уведомления о новых студиях, ссылки на вьюер) | Контракт описан в OpenAPI; тесты на моке Orthanc |
| F9.2 | Задел: входной интерфейс направлений (аналог HL7 ORM) | Контракт описан (схема JSON), реализация — заглушка, документация |
| F9.3 | Задел: исходящий интерфейс заключений (аналог HL7 ORU) | Контракт описан; реализация — позже |

## 6. Модель данных (ключевые сущности)

```
Patient (пациент)
  id, iin (unique), last_name, first_name, middle_name, birth_date,
  gender, phone, email, benefit_category, created_at

  └─< Order (направление)
        id, accession_number (unique), patient_id, referring_physician_id,
        service_id, modality, body_part, priority (ROUTINE/URGENT/CITO),
        diagnosis_icd_id (предв.), clinical_notes, status, created_at

        ├─ Appointment (запись)
        │    id, order_id, device_id, slot_start, slot_end, technologist_id

        ├─ Study (исследование)
        │    id, order_id, study_instance_uid (unique), orthanc_study_id,
        │    technologist_id, acquired_at, qc_status (ACCEPTED/RETAKE)

        └─ Report (заключение)
             id, order_id, radiologist_id, protocol_template_id,
             structured_fields (JSONB), free_text, conclusion,
             critical_finding (boolean), diagnosis_icd_codes (array),
             status (DRAFT/SIGNED/ISSUED), version, parent_report_id,
             signed_at, content_hash, second_opinion_of_report_id
```

**Справочники/системные сущности:**

| Сущность | Ключевые поля |
|---|---|
| Service (тарификатор) | code, name_ru, name_kz, tariff, modality, duration_min, version |
| DiagnosisICD (МКБ-10) | code, name_ru, name_kz, version |
| Device/Modality | name, type (CT/MR/CR/DX/US/MG/RF), ae_title (unique), status |
| ProtocolTemplate | name, modality, body_part, fields_schema (JSONB), version |
| User | id, login, password_hash, role, full_name, is_active |
| AuditLog | id, entity_type, entity_id, action, user_id, timestamp, before_json, after_json |

**Ключевые правила целостности:**
- Order — центральная сущность; без неё нет Appointment, Study, Report.
- Accession Number генерируется на Order, протягивается в MWL → снимки → автосопоставление.
- Report неизменяем после подписи; правка = новая версия (`parent_report_id` → оригинал).
- Second opinion — отдельный Report (`second_opinion_of_report_id`), оригинал не перезаписывается.
- Справочники версионируются: при изменении тарифа/МКБ старые записи не затрагиваются.

## 7. Жизненный цикл исследования (статусы)

### Основная цепочка
```
NEW ──► SCHEDULED ──► ARRIVED ──► IN_PROGRESS ──► ACQUIRED
         MWL уходит              (идёт съёмка)   (снимки в Orthanc
          в Orthanc                                + автосопоставление
                                                   + QC)
                                                      │
                                                      ▼
                                                 TO_REPORT ──► REPORTING ──► SIGNED ──► ISSUED
                                                              (черновик)   (подписано, (выдано;
                                                                            неизменяемо) при critical
                                                                                         → cito)
```

### Боковые переходы

| Переход | Условие | Кто инициирует |
|---|---|---|
| Любой → CANCELLED | До начала исследования (до IN_PROGRESS) | Регистратор / Заведующий |
| ACQUIRED → IN_PROGRESS (переснять) | QC: качество неудовлетворительно | Лаборант |
| TO_REPORT/REPORTING: переназначение | Заведующий меняет исполнителя | Заведующий |
| SIGNED/ISSUED → второе мнение | Создаётся связанный Report | Заведующий |

### Кто управляет переходами

| Переход | Кто меняет |
|---|---|
| NEW → SCHEDULED | Регистратор |
| SCHEDULED → ARRIVED | Регистратор / Лаборант |
| ARRIVED → IN_PROGRESS; QC | Лаборант |
| IN_PROGRESS → ACQUIRED | Система (webhook/поллинг Orthanc) |
| ACQUIRED → TO_REPORT | Система (после QC ACCEPTED) / Лаборант |
| TO_REPORT → REPORTING → SIGNED → ISSUED | Врач-рентгенолог |
| CANCELLED, переназначение, второе мнение | Регистратор / Заведующий |

**Метрики на временных метках статусов:**
- `Turnaround time` = timestamp(ISSUED) − timestamp(NEW)
- `Время ожидания описания` = timestamp(SIGNED) − timestamp(ACQUIRED)
- `Время проведения` = timestamp(ACQUIRED) − timestamp(IN_PROGRESS)

## 8. Интеграции

### 8.1 RIS ↔ Orthanc (DICOM) — реализуется в MVP

**MWL (исходящий):** при переходе в SCHEDULED RIS публикует worklist-запись для
Orthanc MWL-плагина. Обязательные DICOM-теги:

| DICOM тег | Содержимое |
|---|---|
| (0010,0010) PatientName | ФИО пациента |
| (0010,0020) PatientID | ИИН или внутренний ID |
| (0010,0030) PatientBirthDate | Дата рождения YYYYMMDD |
| (0010,0040) PatientSex | M/F/O |
| (0008,0050) AccessionNumber | Accession Number из Order |
| (0020,000D) StudyInstanceUID | Study Instance UID |
| (0008,0060) Modality | CT / MR / CR / DX / US / MG / RF |
| (0040,0001) ScheduledStationAETitle | AE Title аппарата |
| (0040,0002) ScheduledProcedureStepDate | Дата исследования |
| (0040,0003) ScheduledProcedureStepTime | Время исследования |
| (0032,1060) RequestedProcedureDescription | Наименование процедуры |

**Приём снимков (C-STORE, входящий):** аппарат отправляет студию в Orthanc.
RIS узнаёт о новой студии одним из двух способов:

- **Рекомендуется — webhook:** Lua/Python-плагин Orthanc на событие `OnStoredInstance`
  вызывает RIS REST API (реакция в реальном времени, < 5 с).
- **Запасной вариант — REST-поллинг** новых студий через Orthanc REST API раз в 30–60 с
  (если плагины недоступны в окружении). Оба варианта описываются в коде и документации.

**Автосопоставление:** RIS ищет Order по Accession Number / Study Instance UID из тегов
студии → статус ACQUIRED. При несовпадении — очередь ручного связывания (F4.5).

**Просмотр снимков:** из экрана описания открывается встроенный вьюер Orthanc
(Stone Web Viewer / Orthanc Explorer 2) по `study_instance_uid`. Собственный вьюер
не разрабатывается.

### 8.2 ЭЦП — ЭЦП-ready в MVP

- В MVP: подпись = идентификатор врача + timestamp + SHA-256 хэш содержимого заключения +
  блокировка изменений на уровне API.
- Модуль подписи изолирован за интерфейсом `SignatureProvider` — задел под подключение
  **NCALayer / НУЦ РК** (квалифицированная ЭЦП) без изменения бизнес-логики.
- Подписанный документ неизменяем; PDF-заключение содержит реквизиты врача (позже — штамп ЭЦП).

### 8.3 Задел под МИС/HL7 — не реализуется

- Входной контракт направлений (аналог HL7 ORM): JSON-схема описана, в MVP — только
  внутренний UI пользуется этим контрактом.
- Исходящий контракт заключений (аналог HL7 ORU) и нацсистемы РК — контракт описан,
  реализация позже.

## 10. Нефункциональные требования

### 10.1 Безопасность
- RBAC (6 ролей, принцип наименьших привилегий).
- Пароли: хэш bcrypt/argon2; запрет хранения в открытом виде.
- HTTPS/TLS между всеми компонентами (включая RIS ↔ Orthanc).
- Доступ к снимкам только через RIS-сессию/токен (Orthanc не публичен).
- Задел под 2FA (SMS / TOTP).
- Защита от OWASP Top 10: SQL-инъекции, XSS, CSRF, IDOR.

### 10.2 Аудит и целостность данных
- Неизменяемый аудит-лог: хранить навсегда (или не менее срока хранения медзаписей по НПА РК).
- Подписанные заключения неизменяемы на уровне API и БД; правки = новые версии.
- SHA-256 хэш содержимого заключения при подписи; хэш хранится в Report.

### 10.3 Персональные данные и хранение (РК)
- Обработка ПДн согласно Закону РК «О персональных данных и их защите».
- Хранение данных на территории РК (учесть при выборе хостинга).
- Раздельные бэкапы: RIS-БД (PostgreSQL dump) и Orthanc-хранилище (DICOM volume).
- Политика резервного копирования: ежедневный бэкап, хранение не менее 30 дней.

### 10.4 Производительность (ориентиры)
- Открытие рабочего списка: < 2 с при 50 одновременных пользователях.
- Реакция на новый снимок (webhook): < 5 с.
- Поиск пациента: < 1 с.
- Система устойчива к временной недоступности Orthanc: очередь сопоставления не теряется, статусы не зависают.

### 10.5 Локализация и удобство
- i18n-задел: строки UI и документов выделены в ресурсные файлы (ru / kz).
- Форматы дат и чисел по стандартам РК.
- Адаптивный веб-интерфейс (планшет, ноутбук).
- Роле-ориентированные экраны: каждая роль видит только нужный ей набор разделов.

### 10.6 Сопровождаемость
- Чёткие границы модулей (отдельные пакеты/директории).
- Покрытие unit-тестами: машина состояний (статусы заказа), автосопоставление, RBAC.
- Покрытие integration-тестами: полный цикл от создания направления до выдачи заключения.
- OpenAPI-спецификация для REST API (обязательно для интеграционного адаптера).
- Docker Compose для воспроизводимого запуска окружения (RIS + PostgreSQL + Orthanc).
- README с инструкцией по запуску за 5 минут.
