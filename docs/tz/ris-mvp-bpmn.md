### BP3. Проведение исследования (лаборант)

**Роль:** Лаборант

```
НАЧАЛО (день исследования)
  │
  ▼
Открыть рабочий список аппарата
  │  (Список = все SCHEDULED/ARRIVED на этот аппарат сегодня)
  │
  ▼
Пациент прибыл?
  │
  ├─── Нет ──► Ожидать / отметить неявку (CANCELLED)
  │
  ▼ Да
Отметить «Прибыл» → Статус ARRIVED
  │
  ▼
Аппарат запрашивает MWL у Orthanc (C-FIND)
  │  ◄── Автоматически на аппарате
  │
  ▼
Начать исследование на аппарате → Статус IN_PROGRESS
  │
  ▼
Аппарат выполняет исследование
Снимки отправляются в Orthanc (C-STORE)
  │  ◄── Автоматически (DICOM)
  │
  ▼
Orthanc получает студию → webhook → RIS
  │  ◄── Автоматически (< 5 сек)
  │
  ▼
RIS сопоставляет по Accession Number
  │
  ├─── Совпадение ──► Статус ACQUIRED
  │
  └─── Нет совпадения ──► Очередь ручного связывания (F4.5)
                              │
                              ▼
                         Лаборант / Администратор
                         связывает вручную
                              │
                              ▼
                         Статус ACQUIRED
  │
  ▼
Контроль качества снимков
  │
  ├─── Качество OK ──► [Принять] → Статус TO_REPORT
  │
  └─── Качество плохое ──► [Переснять]
                               │
                               ├─ Записать комментарий QC
                               │
                               └─► Статус IN_PROGRESS (новая попытка)
                                   Вернуться к шагу «Начать исследование»
```

### BP4. Описание и выдача заключения

**Роль:** Врач-рентгенолог

```
НАЧАЛО (рабочий список «к описанию»)
  │
  ▼
Выбрать исследование из списка
  │  (Сортировка: cito → срочные → плановые; по времени)
  │
  ▼
Открыть экран описания
  │  ┌─ Данные направления (пациент, услуга, клин. сведения)
  │  └─ Встроенный вьюер Orthanc (снимки по Study UID)
  │
  ▼
Статус → REPORTING
  │
  ▼
Выбрать / подтвердить шаблон протокола
  │  (Подгружается автоматически по модальности / услуге)
  │
  ▼
Заполнить структурированные поля шаблона
  │
  ▼
Написать заключение (свободный текст)
  │
  ▼
Выбрать код(ы) МКБ-10
  │
  ▼
Пометить критическую находку? (critical_finding)
  │
  ├─── Да ──► При выдаче автоматически cito-оповещение направителю
  │
  ▼
[Сохранить черновик] — можно вернуться позже
  │
  ▼
[Подписать заключение]
  │
  ▼
SHA-256 хэш содержимого + timestamp + ID врача → ЭЦП-ready
  │
  ▼
Статус → SIGNED (документ неизменяем)
  │
  ▼
[Выдать заключение]
  │
  ▼
Статус → ISSUED
  │
  ▼
Генерация PDF (реквизиты по НПА РК)
  │
  ├─── critical_finding = true ──► уведомление направителю в системе
  │
  ▼
КОНЕЦ
```

## Часть II. Спецификация форм и полей

### Форма 1. Карточка пациента

**Экран:** Регистратура → Пациент → Создать / Редактировать
**Роли:** Регистратор (создание/редактирование), все роли (просмотр)

| # | Поле | Тип | Обяз. | Валидация / Источник | Примечание |
|---|---|---|---|---|---|
| 1 | ИИН | text | Да | 12 цифр; контрольная цифра по алгоритму МЮ РК | Уникальность; поиск дублей |
| 2 | Фамилия | text | Да | Не пусто; кирилл./латин. | |
| 3 | Имя | text | Да | Не пусто | |
| 4 | Отчество | text | Нет | — | Может отсутствовать |
| 5 | Дата рождения | date | Да | Формат дд.мм.гггг; не будущее | |
| 6 | Пол | select | Да | М / Ж | |
| 7 | Телефон | text | Нет | +7 (XXX) XXX-XX-XX | |
| 8 | E-mail | text | Нет | Формат email | |
| 9 | Льготная категория | select | Нет | ГОБМП / ОСМС / Инвалидность / Нет | Задел под тарификацию |
| 10 | Документ | select | Нет | Удостоверение / Паспорт | Задел для иностранцев |
| 11 | Номер документа | text | Нет | Зависит от типа документа | |
| 12 | Комментарий | textarea | Нет | До 500 символов | |

**Дополнительно на карточке (только чтение):**
- История исследований (хронологический список направлений)
- Дата создания и последнего обновления
- Кто создал / обновил (из аудит-лога)

### Форма 2. Направление (Order)

**Экран:** Регистратура → Направления → Создать
**Роль:** Регистратор, Врач-направитель (только свои пациенты)

| # | Поле | Тип | Обяз. | Валидация / Источник | Примечание |
|---|---|---|---|---|---|
| 1 | Пациент | autocomplete | Да | Поиск по ИИН / ФИО | Ссылка на Patient.id |
| 2 | Услуга | autocomplete | Да | Справочник: тарификатор МЗ РК | Фильтр по модальности |
| 3 | Модальность | select (auto) | Да | Заполняется из услуги | CT / MR / CR / DX / US / MG / RF |
| 4 | Область исследования | text / select | Нет | Свободный текст или справочник | Например: «Грудная клетка» |
| 5 | Тип финансирования | select | Да | ГОБМП / ОСМС / Платно | Задел под ФОМС-отчётность |
| 6 | Приоритет | select | Да | Плановый / Срочный / Cito | Cito — подсветка красным |
| 7 | Врач-направитель | autocomplete | Да | Справочник пользователей (роль: врач-направитель) | Может быть текущий пользователь |
| 8 | Предварительный диагноз | autocomplete | Нет | Справочник МКБ-10 (код + наим.) | |
| 9 | Клинические сведения | textarea | Нет | До 2000 символов | Анамнез, жалобы |
| 10 | Контрастное вещество | checkbox | Нет | — | Флаг; для КТ/МРТ |
| 11 | Комментарий | textarea | Нет | До 500 символов | |

**Генерируется автоматически (только чтение):**
- Accession Number (уникальный)
- Дата/время создания направления
- Статус: NEW

### Форма 3. Запись в расписание (Appointment)

**Экран:** Регистратура → Расписание → Календарь
**Роль:** Регистратор

| # | Поле | Тип | Обяз. | Валидация / Источник | Примечание |
|---|---|---|---|---|---|
| 1 | Направление | auto (из BP2) | Да | Передаётся из формы направления | AN, пациент, услуга |
| 2 | Аппарат | select | Да | Справочник аппаратов; фильтр по модальности | Только аппараты со статусом «Активен» |
| 3 | Дата | datepicker | Да | Не в прошлом; рабочие дни аппарата | Из календаря |
| 4 | Слот времени | select | Да | Только свободные слоты выбранного аппарата | Длительность из услуги |
| 5 | Лаборант | select | Нет | Справочник пользователей (роль: лаборант) | Предзаполнение по умолчанию |
| 6 | Комментарий | textarea | Нет | До 300 символов | Для лаборанта |

**После сохранения:**
- Статус направления: SCHEDULED
- MWL-запись автоматически публикуется в Orthanc

### Форма 5. Экран описания и заключения

**Экран:** Описание → Выбрать исследование → Открыть
**Роль:** Врач-рентгенолог

**Панель левая — данные исследования (только чтение):**

| Поле | Источник |
|---|---|
| ФИО пациента | Patient |
| Дата рождения / Пол / ИИН | Patient |
| Услуга / Модальность | Order |
| Предв. диагноз (МКБ-10) | Order |
| Клинические сведения | Order |
| Врач-направитель | Order |
| Приоритет / Тип финансирования | Order |
| Accession Number | Order |
| Дата / время исследования | Study.acquired_at |
| Лаборант | Study |
| История предыдущих исследований | Patient history |

**Панель правая — вьюер Orthanc:**
- Stone Web Viewer / Orthanc Explorer 2 встроен через iframe / deeplink по `study_instance_uid`
- Инструменты вьюера: панель/окно, измерения, аннотации — в самом Orthanc

**Форма описания (центр):**

| # | Поле | Тип | Обяз. | Валидация / Источник | Примечание |
|---|---|---|---|---|---|
| 1 | Шаблон протокола | autocomplete | Нет | Справочник шаблонов; фильтр по модальности/области | Автовыбор по модальности услуги |
| 2 | Структурированные поля | dynamic form | Нет | Из fields_schema шаблона (JSONB) | Зависят от шаблона: radio, select, textarea, number |
| 3 | Описание (свободный текст) | textarea | Нет | До 10 000 символов | Markdown/plain text |
| 4 | Заключение | textarea | Да | До 3000 символов | Итоговый вывод |
| 5 | Диагноз по МКБ-10 | autocomplete (multiple) | Да | Справочник МКБ-10; минимум 1 | Основной + сопутствующие |
| 6 | Критическая находка | checkbox | Нет | — | При выдаче → cito-оповещение |

## Часть III. Интеграционный задел (MVP без реализации)

### Принцип изоляции

Все внешние интеграции в MVP изолированы за единым `IntegrationAdapter` — фасадом с чётко описанным JSON-контрактом. В MVP адаптер существует как заглушка (`stub`). Реальная реализация подключается без изменения бизнес-логики.

```
RIS Core (бизнес-логика)
       │
       ▼
IntegrationAdapter (интерфейс)
       │
   ┌───┴───────────────────────────────────────┐
   │                                           │
   ▼                                           ▼
OrthancAdapter (реализован в MVP)     ExternalAdapter (stub в MVP)
   │                                           │
   ▼                                   ┌───────┼───────────┐
Orthanc REST/DICOMweb                  │       │           │
                                      HL7    МИС РК     ОСМС
                                     Adapter  Adapter   Adapter
                                    (будущее) (будущее) (будущее)
```

### Задел 1. Входящие направления — аналог HL7 ORM

**Назначение:** получение электронных направлений из внешней МИС (поликлиника, ОСМС-портал).

**Эндпоинт (заглушка в MVP):**
```
POST /api/v1/adapter/orders/inbound
Content-Type: application/json
Authorization: Bearer <api_key>
```

**Контракт запроса:**
```json
{
  "source_system": "MIS_NADEZHDA",
  "source_order_id": "EXT-2026-001234",
  "patient": {
    "iin": "900101300012",
    "last_name": "Иванов",
    "first_name": "Иван",
    "middle_name": "Иванович",
    "birth_date": "1990-01-01",
    "gender": "M",
    "phone": "+77001234567"
  },
  "order": {
    "service_code": "A06.10.001",
    "modality": "CT",
    "body_part": "Грудная клетка",
    "priority": "ROUTINE",
    "financing_type": "OSMS",
    "referring_physician": {
      "full_name": "Петров А.А.",
      "organization": "Поликлиника №5"
    },
    "diagnosis_icd": "J18.9",
    "clinical_notes": "Кашель 2 недели, субфебрилитет.",
    "requested_date": "2026-06-15"
  }
}
```

**Контракт ответа:**
```json
{
  "status": "ACCEPTED",
  "ris_order_id": "RIS-2026-00456",
  "accession_number": "260613-00456",
  "message": "Направление принято и зарегистрировано"
}
```

**Поведение MVP:** эндпоинт возвращает `501 Not Implemented` с телом
`{"status":"STUB","message":"External order intake not implemented in MVP"}`.
Контракт задокументирован в OpenAPI. Все входящие направления в MVP создаются вручную через UI.

### Задел 2. Исходящие заключения — аналог HL7 ORU

**Назначение:** отправка подписанных заключений во внешние системы (МИС, РЦЭЗ).

**Эндпоинт (заглушка в MVP):**
```
Событие: Report.status → ISSUED
Действие: IntegrationAdapter.sendReportOutbound(report_id)
```

**Контракт события (JSONB, хранится в AuditLog при ISSUED):**
```json
{
  "event": "REPORT_ISSUED",
  "ris_order_id": "RIS-2026-00456",
  "accession_number": "260613-00456",
  "patient_iin": "900101300012",
  "report": {
    "id": "RPT-2026-00789",
    "radiologist": "Сидоров В.В.",
    "signed_at": "2026-06-13T10:30:00+06:00",
    "diagnosis_icd": ["J18.9"],
    "conclusion": "Пневмония нижней доли правого лёгкого.",
    "critical_finding": false,
    "content_hash": "sha256:abc123...",
    "pdf_url": "/api/v1/reports/RPT-2026-00789/pdf"
  }
}
```

**Поведение MVP:** событие логируется в AuditLog, во внешнюю систему не отправляется.
При реализации — вызов webhook внешней системы или HL7 ORU-сообщение через адаптер.

### Задел 3. Нацсистемы РК — РЦЭЗ / МИС «Надежда»

**Контекст:** Закон РК «Об информатизации» обязывает подключение к ШЭП (шлюзу электронного правительства) при обмене данными с госсистемами. Это требует:
- Сертификата НУЦ РК на сервере
- Регистрации ИС в РЦЭЗ
- Прохождения тестирования интеграции

**Задел в MVP:**
1. `Organization.external_id_rcez` — поле для хранения идентификатора в РЦЭЗ.
2. `Order.external_id_mis` — поле для хранения ID направления в МИС «Надежда».
3. Экспорт вручную: администратор может выгрузить реестр исследований в XML/CSV по форме РЦЭЗ.

**Этапы подключения (после MVP):**
```
Шаг 1: Регистрация организации в РЦЭЗ
Шаг 2: Регистрация ИС RIS в реестре МИС МЗ РК
Шаг 3: Получение сертификата НУЦ РК для сервера
Шаг 4: Настройка ШЭП-шлюза
Шаг 5: Тестирование обмена в тестовой среде РЦЭЗ
Шаг 6: Продуктивное подключение
```

---

### Задел 4. ОСМС — тип финансирования и будущая отчётность

**Контекст:** Частные ДЦ, работающие по ОСМС, ежемесячно подают реестры в ФОМС РК.

**Задел в MVP:**
- Поле `Order.financing_type` (GOMBP / OSMS / PAID) хранится для каждого заказа.
- Отчёт F8.5 «по типам финансирования» строится уже в MVP.
- Формат реестра ФОМС РК — не реализуется в MVP, реализуется через `ExternalAdapter`.

**Экспорт вручную (MVP):** CSV с полями: AN, дата, ИИН, код услуги ОСМС, тариф, врач.

---

### Задел 5. ЭЦП НУЦ РК — квалифицированная подпись

**Контекст:** Для юридически значимых медицинских документов в РК требуется квалифицированная ЭЦП через НУЦ РК / NCALayer.

**Задел в MVP:**
```
SignatureProvider (интерфейс)
       │
   ┌───┴──────────────────────────┐
   │                              │
   ▼                              ▼
SimpleSignatureProvider    NCALayerSignatureProvider
(MVP: врач + timestamp      (будущее: реальная ЭЦП
 + SHA-256 + блокировка)     через браузерный плагин
                             NCALayer)
```

**Поля, которые добавляются при переходе на НУЦ РК:**
- `Report.ecds_signature` (PKCS#7 / CMS подпись)
- `Report.certificate_serial` (серийный номер сертификата)
- `Report.certificate_iin` (ИИН владельца сертификата — должен совпадать с врачом)

**Переход с MVP на НУЦ РК:** замена `SimpleSignatureProvider` на `NCALayerSignatureProvider` без изменения бизнес-логики. Контракт интерфейса:
```
sign(document_hash: str, user_id: int) → SignatureResult
verify(document_hash: str, signature: bytes) → VerifyResult
```
