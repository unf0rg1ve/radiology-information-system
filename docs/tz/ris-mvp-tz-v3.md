## 0. Рыночный контекст и позиционирование

### Проблема рынка

Рынок RIS в Казахстане не обслужен в сегменте частных диагностических центров (ДЦ).

| Сегмент | Что предлагают | Почему не подходит частному ДЦ |
|---|---|---|
| **Международные** (GE Centricity, Siemens syngo) | Полнофункциональные enterprise RIS/PACS | Цена и внедрение недоступны для ДЦ 2–10 аппаратов; нет локализации под РК |
| **Российские RIS** (ЕРИС, iRIS, ЛИНС) | Зрелые RIS с DICOM | Нет тарификатора МЗ РК, нет ЭЦП НУЦ РК, нет казахского языка; риски после 2022 г. |
| **Казахстанские МИС** (1С:Медицина, Smart Delta, MEDODS) | Регистратура, расписание, ЭМК | Нет radiology workflow: MWL, C-STORE, описание по шаблонам, подпись |
| **PACS-дистрибьюторы** (МСТ Казахстан, Интер-Мед) | Хранение и просмотр DICOM | Только архив снимков, нет RIS-процесса |
| **Гос. МИС «Надежда»** | Для государственных клиник | Не предназначена для частных ДЦ; нет специализированного RIS-модуля |

**Свободная ниша:** частный диагностический центр в РК с 2–10 аппаратами — полный radiology workflow с казахстанской спецификой по цене SaaS-подписки.

### Целевой сегмент MVP

- **Первичный:** частные диагностические центры (КТ, МРТ, рентген), 2–10 модальностей, 5–30 сотрудников. Города: Алматы, Астана (Нур-Султан), Шымкент.
- **Вторичный:** отделения лучевой диагностики в небольших частных клиниках и многопрофильных медицинских центрах.
- **Отложено:** государственные ЛПУ, крупные сети (10+ точек), телерадиологические сервисы.

### Конкурентное преимущество MVP

1. Единственный RIS с нативной поддержкой тарификатора МЗ РК (ГОБМП/ОСМС).
2. ЭЦП-ready под НУЦ РК — путь к квалифицированной подписи без переписывания.
3. Двуязычный интерфейс и документы (RU + KZ).
4. Соответствие минимальным требованиям к МИС по НПА РК.
5. On-premise + SaaS в ЦОД РК — данные не покидают страну.
6. Цена и скорость внедрения, недостижимые для международных вендоров.

## 1. Назначение и контекст системы

Радиологическая информационная система (RIS) автоматизирует рабочий процесс отделения
лучевой диагностики: от регистрации пациента и направления — до проведения исследования,
описания врачом-рентгенологом и выдачи заключения.

**Классический поток:**
```
Регистрация пациента
      ↓
Создание направления (услуга, приоритет, МКБ-10)
      ↓
Запись в расписание (аппарат, слот времени)
      ↓
Передача задания на аппарат через DICOM MWL
      ↓
Проведение исследования → приём снимков (C-STORE → Orthanc)
      ↓
Контроль качества снимков (лаборант)
      ↓
Описание по шаблону протокола (врач-рентгенолог)
      ↓
Подпись и выдача заключения (PDF + ЭЦП-ready)
      ↓
Статистика, метрики, отчётность
```

**Контекст MVP:** самостоятельная (standalone) система для частного диагностического центра.
Архитектура **integration-ready** — приём направлений изолирован за адаптером, чтобы
позже добавить HL7/МИС и нацсистемы РК без переписывания ядра.

### Ориентиры (best practices)
- **GE Centricity RIS** — структурированные шаблоны протоколов, метрики turnaround time.
- **Agfa Enterprise Imaging** — задача-ориентированный workflow: исследование = набор задач (QC → описание → подпись).
- **Orthanc / ThaiRIS** — зрелые open-source DICOM-компоненты как база.

## 2. Границы MVP

### В составе MVP
- Регистратура: поиск, создание и редактирование карточки пациента.
- Направления: создание, управление статусами, печать (форма по НПА РК).
- Расписание: календарь по аппаратам, запись, отметка явки.
- DICOM Modality Worklist (MWL), приём снимков (C-STORE), просмотр снимков через Orthanc.
- Описание исследований по шаблонам, подпись (ЭЦП-ready), выдача заключений в PDF.
- 6 ролей с ролевым доступом (RBAC).
- Справочники РК: тарификатор услуг (ГОБМП/ОСМС), МКБ-10, аппараты, шаблоны протоколов.
- Аудит всех действий с медданными; версионирование заключений.
- Статистика и метрики (turnaround time, нагрузка на аппараты и врачей).
- cito-оповещения направителю при критической находке.
- Второе мнение / рецензия (инициирует заведующий).

### Вне MVP (отложено, архитектурный задел сохранён)
- Личный кабинет пациента и SMS-оповещения пациенту.
- Полноценный биллинг и ОСМС-портал (тарификатор используется как справочник с тарифом).
- Интеграция с МИС по HL7 v2.x (только контракт адаптера).
- Отчётность в нацсистемы РК (РЦЭЗ, МИС «Надежда», ЕДСЗН).
- Казахская локализация контента (i18n-задел в архитектуре есть; перевод позже).
- Квалифицированная ЭЦП через NCALayer / НУЦ РК (модуль подписи изолирован, интеграция позже).
- Телерадиология: удалённое описание снимков внешним радиологом (задел через адаптер).
- Многофилиальность: сети ДЦ с несколькими точками (задел в модели данных — `organization_id`).
- SaaS / multi-tenant (MVP — single-tenant on-premise или VPS).

## 3. Архитектура

**Вариант A — модульный монолит + Orthanc как отдельный DICOM-узел.**

```
┌─────────────────────────────────────────────────────────────┐
│              Веб-фронтенд (SPA)                             │
│        Рабочие места 6 ролей (React / Vue)                  │
└────────────────────────┬────────────────────────────────────┘
                         │  REST/JSON + WebSocket
┌────────────────────────▼────────────────────────────────────┐
│              RIS Backend (модульный монолит)                 │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────────┐   │
│  │Пациенты/ │ │Направл./ │ │Описания/ │ │Пользователи/ │   │
│  │регистр.  │ │расписание│ │подпись   │ │роли / аудит  │   │
│  └──────────┘ └──────────┘ └──────────┘ └──────────────┘   │
│  ┌──────────┐ ┌────────────────────────────────────────┐    │
│  │Справочн. │ │Статистика / Метрики / Интеграц. адаптер│    │
│  └──────────┘ └────────────────────────────────────────┘    │
└──────┬──────────────────────────────────────┬───────────────┘
       │  SQL                                 │  DICOMweb / REST
┌──────▼──────┐                    ┌──────────▼──────────────┐
│ PostgreSQL  │                    │        Orthanc           │
│   (RIS DB)  │                    │  MWL · C-STORE · вьюер  │
└─────────────┘                    └──────────┬──────────────┘
                                              │  DICOM
                                   КТ · МРТ · РГ · УЗИ · ММГ
```

### Принципы архитектуры

| Принцип | Описание |
|---|---|
| Разделение ответственности | RIS — рабочий процесс и данные; Orthanc — DICOM. RIS не дублирует пиксели, ссылается по идентификаторам |
| Связка по Accession Number | AN протягивается через весь цикл: заказ → MWL → снимки → автосопоставление |
| Интеграционный адаптер | Изолирует внешний мир (сейчас Orthanc; задел — HL7, нацсистемы РК, МИС «Надежда») |
| Сквозной аудит | Все действия с медданными фиксируются в неизменяемом лог-журнале |
| Задел на многофилиальность | Поле `organization_id` во всех ключевых сущностях; в MVP используется единственная организация |
| Чёткие границы модулей | Каждый модуль тестируется независимо; задел под выделение в сервисы при росте |

**Деплой-топология MVP:** Docker Compose (3 контейнера: RIS backend, PostgreSQL, Orthanc).
Может запускаться on-premise на сервере клиники или на VPS в ЦОД РК (Казтелеком, KDDI).

## 4. Роли и рабочие места

| Роль | Основные функции |
|---|---|
| **Регистратор** | Поиск/создание пациента; создание направления; запись в расписание; отметка явки; печать направления/талона |
| **Рентгенолаборант** | Рабочий список аппарата; статусы «прибыл / начато / снимки получены»; контроль качества (принять / переснять) |
| **Врач-рентгенолог** | Рабочий список «к описанию»; просмотр снимков (вьюер Orthanc); описание по шаблону + МКБ-10; подпись и выдача заключения |
| **Заведующий отделением** | Обзор нагрузки и статусов; переназначение описаний; инициация второго мнения; контроль просроченных |
| **Врач-направитель** | Создание направлений своим пациентам; просмотр статуса и готовых заключений (только чтение, только свои) |
| **Администратор** | Пользователи и роли; справочники; аудит; системные настройки; регистрация организации |

**Доступ:** RBAC, принцип наименьших привилегий. Доступ к снимкам — только через RIS-сессию.

## 5. Функциональные требования

### M1. Пациенты и регистратура

| ID | Требование | Критерии приёмки |
|---|---|---|
| F1.1 | Поиск по ИИН / ФИО / дате рождения / телефону | Результат при вводе ≥ 3 символов; нечёткий поиск по ФИО |
| F1.2 | Создание/редактирование карточки: ИИН, ФИО, пол, ДР, контакты, льготная категория (ГОБМП/ОСМС) | ИИН валидируется (12 цифр + контрольная цифра); обязательные поля по НПА РК |
| F1.3 | Контроль дублей по ИИН | Предупреждение со ссылкой на существующего пациента при совпадении ИИН |
| F1.4 | История исследований пациента | Хронологический список направлений и заключений с фильтром по периоду |

### M2. Направления (orders)

| ID | Требование | Критерии приёмки |
|---|---|---|
| F2.1 | Создание: пациент → услуга (тарификатор) → модальность → область → врач-направитель → приоритет (плановый / срочный / cito) → предв. диагноз (МКБ-10) → клинические сведения | Все обязательные поля заполнены; направление сохраняется в статусе NEW |
| F2.2 | Генерация Accession Number и идентификатора заказа | AN уникален; формат настраивается администратором |
| F2.3 | Управление статусами (см. раздел 7) | Переходы только по разрешённым дугам; нарушение — ошибка 403 |
| F2.4 | Отмена с указанием причины | Доступна до начала исследования; причина обязательна |
| F2.5 | Печать направления | PDF с реквизитами по НПА РК (организация, врач, пациент, ИИН, услуга, МКБ, дата) |
| F2.6 | Тип финансирования: ГОБМП / ОСМС / платно | Поле хранится в заказе; задел под интеграцию с порталом ОСМС |

### M3. Расписание и запись

| ID | Требование | Критерии приёмки |
|---|---|---|
| F3.1 | Календарь по аппаратам (день/неделя), слоты по длительности услуги | Слоты формируются автоматически по рабочему времени аппарата и длительности |
| F3.2 | Запись в слот; перенос/отмена | Занятый слот недоступен; при переносе — старый освобождается |
| F3.3 | Учёт недоступности аппарата (ТО, поломка) | Администратор устанавливает окна; слоты в это время не предлагаются |
| F3.4 | Отметка явки пациента | Статус SCHEDULED → ARRIVED |
| F3.5 | Предупреждение о конфликтах / двойной записи | Система не позволяет записать один аппарат дважды в один слот |

### M4. Worklist и проведение исследования

| ID | Требование | Критерии приёмки |
|---|---|---|
| F4.1 | Публикация в DICOM MWL: демография + AN + Study UID + AE Title | Аппарат получает запись через C-FIND; данные совпадают с направлением |
| F4.2 | Рабочий список лаборанта по аппарату | Только исследования текущего аппарата; сортировка по времени записи |
| F4.3 | Статусы исследования с временными метками | Переходы фиксируются; доступны только лаборанту своего аппарата |
| F4.4 | Автосопоставление студии с заказом по AN / Study UID | Статус → ACQUIRED автоматически, < 5 с после webhook |
| F4.5 | Ручное связывание при расхождении | Очередь «не сопоставлено»; лаборант / администратор связывает вручную |
| F4.6 | Контроль качества: принять / переснять + комментарий | «Переснять» возвращает IN_PROGRESS; попытка сохраняется в истории |

### M5. Описание и заключение

| ID | Требование | Критерии приёмки |
|---|---|---|
| F5.1 | Рабочий список «к описанию» с фильтрами | cito наверху; фильтры без перезагрузки страницы |
| F5.2 | Экран описания: данные направления + вьюер Orthanc | Вьюер открывается по study_instance_uid в том же экране |
| F5.3 | Шаблоны протоколов: структурированные поля + свободный текст | Шаблон подгружается по модальности; поля редактируемы |
| F5.4 | Заключение + код(ы) МКБ-10 | Поиск по коду и наименованию; минимум один код обязателен |
| F5.5 | Черновик → подпись (ЭЦП-ready: врач + timestamp + SHA-256) | После подписи недоступен для редактирования; хэш верифицируем |
| F5.6 | Выдача заключения (PDF); при critical_finding → cito-оповещение направителю | PDF с реквизитами по НПА РК; уведомление видно в интерфейсе |
| F5.7 | Второе мнение (инициирует заведующий) | Отдельный Report; оригинал не перезаписывается |

### M6. Справочники

| ID | Требование | Критерии приёмки |
|---|---|---|
| F6.1 | Тарификатор услуг (код ГОБМП/ОСМС, наименование, тариф, модальность, длительность) | Импорт Excel/CSV; версионирование; изменения не затрагивают старые записи |
| F6.2 | МКБ-10 | Полный справочник; поиск по частичному совпадению кода и наименования |
| F6.3 | Аппараты/модальности (тип, AE Title, расписание, статус) | AE Title уникален; блокировка записи при отключении аппарата |
| F6.4 | Шаблоны протоколов: создание/редактирование администратором | Привязка к услуге/модальности; версионирование |
| F6.5 | Пользователи, роли, права | Смена пароля, блокировка, журнал входов |
| F6.6 | Реквизиты организации | Наименование, лицензия МЗ РК, адрес, контакты — выводятся в печатных формах |

### M7. Пользователи, доступ, аудит

| ID | Требование | Критерии приёмки |
|---|---|---|
| F7.1 | Аутентификация (логин/пароль; задел под 2FA) | Пароль — bcrypt/argon2; сессия завершается по таймауту |
| F7.2 | RBAC: 6 ролей | Попытка получить ресурс без прав → 403; врач-направитель видит только своих пациентов |
| F7.3 | Сквозной аудит-лог | Неизменяем; entity, action, user_id, timestamp, before/after |
| F7.4 | Неизменяемость подписанных документов | Редактирование подписанного → отклонение API; правка = новая версия |

### M8. Статистика и метрики

| ID | Требование | Критерии приёмки |
|---|---|---|
| F8.1 | Дашборд: исследования по периодам / модальностям / врачам / аппаратам | Фильтры по дате и модальности работают |
| F8.2 | Turnaround time (ISSUED − NEW); доля просроченных | Метрика корректна; настраиваемый порог «просрочено» |
| F8.3 | Нагрузка на аппараты и врачей | Число исследований за период по каждому ресурсу |
| F8.4 | Выгрузка в CSV/Excel | Файл содержит все строки выборки; UTF-8 |
| F8.5 | Отчёт по типам финансирования (ГОБМП / ОСМС / платно) | Разбивка по источнику оплаты за период; задел под отчёт в ФОМС |

### M9. Интеграционный адаптер

| ID | Требование | Критерии приёмки |
|---|---|---|
| F9.1 | DICOMweb/REST-обмен с Orthanc (MWL, webhook/поллинг, ссылки на вьюер) | Контракт в OpenAPI; тесты на моке Orthanc |
| F9.2 | Задел: входной интерфейс направлений (аналог HL7 ORM) | JSON-схема описана; в MVP — только внутренний UI; заглушка с документацией |
| F9.3 | Задел: исходящий интерфейс заключений (аналог HL7 ORU) | Контракт описан; реализация — позже |
| F9.4 | Задел: отчётность в нацсистемы РК (РЦЭЗ, МИС «Надежда») | Контракт описан; в MVP — экспорт CSV/XML вручную; интеграция — позже |

## 6. Модель данных (ключевые сущности)

```
Organization (организация — задел на многофилиальность)
  id, name_ru, name_kz, license_number, address, phone

Patient (пациент)
  id, organization_id*, iin (unique), last_name, first_name, middle_name,
  birth_date, gender, phone, email, benefit_category (GOMBP/OSMS/NONE)

  └─< Order (направление)
        id, accession_number (unique), patient_id, organization_id*,
        referring_physician_id, service_id, modality, body_part,
        priority (ROUTINE/URGENT/CITO), financing_type (GOMBP/OSMS/PAID),
        diagnosis_icd_id, clinical_notes, status, created_at

        ├─ Appointment (запись)
        │    id, order_id, device_id, slot_start, slot_end, technologist_id

        ├─ Study (исследование)
        │    id, order_id, study_instance_uid (unique), orthanc_study_id,
        │    technologist_id, acquired_at, qc_status (ACCEPTED/RETAKE),
        │    qc_attempts (JSONB — история попыток)

        └─ Report (заключение)
             id, order_id, radiologist_id, protocol_template_id,
             structured_fields (JSONB), free_text, conclusion,
             critical_finding (bool), diagnosis_icd_codes (array),
             status (DRAFT/SIGNED/ISSUED), version, parent_report_id,
             signed_at, content_hash (SHA-256), second_opinion_of_report_id
```

`*` — `organization_id` добавляется во все сущности как задел на многофилиальность;
в MVP единственная организация, фильтрация по ней не требуется.

**Справочники:**

| Сущность | Ключевые поля |
|---|---|
| Service (тарификатор) | code_gombp, code_osms, name_ru, name_kz, tariff, modality, duration_min, version, valid_from |
| DiagnosisICD | code, name_ru, name_kz, version |
| Device | name, type (CT/MR/CR/DX/US/MG/RF), ae_title (unique), org_id*, schedule, status |
| ProtocolTemplate | name_ru, modality, body_part, fields_schema (JSONB), version |
| User | id, org_id*, login, password_hash, role, full_name, is_active |
| AuditLog | entity_type, entity_id, action, user_id, timestamp, before_json, after_json |

**Правила целостности:**
- Order — центральная сущность; без неё нет Appointment, Study, Report.
- Accession Number генерируется на Order, протягивается в MWL → снимки → автосопоставление.
- Report неизменяем после подписи; правка = новая версия (`parent_report_id`).
- Second opinion — отдельный Report (`second_opinion_of_report_id`), оригинал не перезаписывается.
- Справочники версионируются; при изменении тарифа/МКБ старые заказы не затрагиваются.

## 7. Жизненный цикл исследования

### Основная цепочка
```
NEW ──► SCHEDULED ──► ARRIVED ──► IN_PROGRESS ──► ACQUIRED
        (MWL → Orthanc)           (идёт съёмка)   (снимки + QC)
                                                       │
                                                  TO_REPORT ──► REPORTING ──► SIGNED ──► ISSUED
                                                               (черновик)   (неизменяемо) (при critical
                                                                                           → cito)
```

### Боковые переходы

| Переход | Условие | Инициатор |
|---|---|---|
| Любой → CANCELLED | До IN_PROGRESS | Регистратор / Заведующий |
| ACQUIRED → IN_PROGRESS (переснять) | QC: качество неудовлетворительно | Лаборант |
| TO_REPORT/REPORTING: смена исполнителя | Переназначение | Заведующий |
| SIGNED/ISSUED → второе мнение | Создаётся связанный Report | Заведующий |

### Таблица переходов

| Переход | Инициатор |
|---|---|
| NEW → SCHEDULED | Регистратор |
| SCHEDULED → ARRIVED | Регистратор / Лаборант |
| ARRIVED → IN_PROGRESS; QC | Лаборант |
| IN_PROGRESS → ACQUIRED | Система (webhook Orthanc < 5 с) |
| ACQUIRED → TO_REPORT | Система (после QC ACCEPTED) |
| TO_REPORT → REPORTING → SIGNED → ISSUED | Врач-рентгенолог |
| CANCELLED, переназначение, второе мнение | Регистратор / Заведующий |

**Метрики на временных метках:**
- Turnaround time = `timestamp(ISSUED)` − `timestamp(NEW)`
- Время ожидания описания = `timestamp(SIGNED)` − `timestamp(ACQUIRED)`
- Время проведения = `timestamp(ACQUIRED)` − `timestamp(IN_PROGRESS)`

## 8. Интеграции

### 8.1 RIS ↔ Orthanc (DICOM) — в MVP

**MWL (исходящий):** при SCHEDULED RIS публикует запись для Orthanc MWL-плагина.
Обязательные DICOM-теги:

| DICOM тег | Содержимое |
|---|---|
| (0010,0010) PatientName | ФИО пациента |
| (0010,0020) PatientID | ИИН или внутренний ID |
| (0010,0030) PatientBirthDate | YYYYMMDD |
| (0010,0040) PatientSex | M / F / O |
| (0008,0050) AccessionNumber | Accession Number из Order |
| (0020,000D) StudyInstanceUID | Study Instance UID |
| (0008,0060) Modality | CT / MR / CR / DX / US / MG / RF |
| (0040,0001) ScheduledStationAETitle | AE Title аппарата |
| (0040,0002/0003) ScheduledProcedureStepDate/Time | Дата и время |
| (0032,1060) RequestedProcedureDescription | Наименование процедуры |

**Приём снимков:** аппарат отправляет студию в Orthanc (C-STORE). RIS узнаёт:
- **Рекомендуется — webhook:** Lua/Python-плагин Orthanc `OnStoredInstance` → RIS REST API, < 5 с.
- **Запасной — REST-поллинг:** опрос Orthanc REST API раз в 30–60 с.

**Просмотр:** Stone Web Viewer / Orthanc Explorer 2 по `study_instance_uid`. Собственный вьюер не разрабатывается.

### 8.2 ЭЦП — ЭЦП-ready в MVP

- **MVP:** врач + timestamp + SHA-256 хэш содержимого + блокировка на уровне API.
- Модуль подписи изолирован за интерфейсом `SignatureProvider` — задел под **NCALayer / НУЦ РК** без изменения бизнес-логики.
- Подписанный PDF содержит реквизиты врача; позже — штамп ЭЦП НУЦ РК.

### 8.3 МИС/HL7 — задел, не реализуется

- Входной контракт направлений (HL7 ORM-аналог): JSON-схема описана; в MVP — только внутренний UI.
- Исходящий контракт заключений (HL7 ORU-аналог) и нацсистемы РК — позже.

### 8.4 Нацсистемы РК — задел

- Взаимодействие с МИС «Надежда» / РЦЭЗ: задел через интеграционный адаптер; в MVP — ручной экспорт XML/CSV.
- Подключение через ШЭП (шлюз «электронного правительства») — требование Закона РК об информатизации; реализуется после MVP.
- ОСМС-портал (ФОМС РК): задел на отправку реестров; поле `financing_type` в заказе хранит тип финансирования.

## 10. Нефункциональные требования

### 10.1 Безопасность
- RBAC (6 ролей, принцип наименьших привилегий).
- Пароли: bcrypt/argon2; запрет хранения в открытом виде.
- HTTPS/TLS между всеми компонентами (включая RIS ↔ Orthanc).
- Доступ к снимкам только через RIS-сессию/токен (Orthanc закрыт от прямого доступа).
- Задел под 2FA (TOTP / SMS).
- Защита от OWASP Top 10: SQL-инъекции, XSS, CSRF, IDOR.
- JWT-токены с коротким сроком жизни; refresh-токен — httpOnly cookie.

### 10.2 Аудит и целостность
- Неизменяемый аудит-лог: хранить не менее срока хранения медзаписей по НПА РК (25 лет для взрослых).
- Подписанные заключения неизменяемы на уровне API и БД; правки = новые версии.
- SHA-256 хэш содержимого заключения при подписи; хэш хранится в Report.

### 10.3 Персональные данные и хранение (РК)
- Обработка ПДн: Закон РК **«О персональных данных и их защите»** № 94-V от 21.05.2013.
- **Локализация данных:** хранение персональных данных граждан РК на территории РК (ЦОД Казтелеком / KDDI / собственный сервер клиники).
- Раздельные бэкапы: RIS-БД (PostgreSQL dump) и Orthanc (DICOM volume).
- Ежедневный бэкап; хранение не менее 30 дней.

### 10.4 Производительность (ориентиры MVP)
- Открытие рабочего списка: < 2 с при 20–50 одновременных пользователях.
- Реакция на новый снимок (webhook): < 5 с.
- Поиск пациента: < 1 с.
- Устойчивость к временной недоступности Orthanc: очередь сопоставления не теряется.

### 10.5 Локализация и удобство
- i18n-задел: все строки UI и документов выделены в ресурсные файлы (ru / kz).
- Форматы дат и чисел по стандартам РК (дд.мм.гггг, казахские символы UTF-8).
- Адаптивный веб-интерфейс (планшет, ноутбук; мобильный — задел).
- Роле-ориентированные экраны: каждая роль видит только нужный набор разделов.

### 10.6 Соответствие НПА РК
- Приказ МЗ РК **№ ҚР ДСМ-175 (V2100023926)** — минимальные требования к МИС.
- Печатные формы: реквизиты организации (лицензия МЗ РК), врача, пациента (ИИН), услуги (код тарификатора), МКБ-10, дата, подпись.
- Закон РК **«Об информатизации»** № 418-V — требования к ИС государственных органов и взаимодействию с ними.
- Кодекс РК **«О здоровье народа и системе здравоохранения»** № 360-VI — требования к МИС и электронным медицинским записям.

### 10.7 Сопровождаемость
- Чёткие границы модулей (отдельные пакеты/директории).
- Unit-тесты: машина состояний (статусы), автосопоставление, RBAC.
- Integration-тесты: полный цикл создание направления → выдача заключения.
- OpenAPI-спецификация для REST API (обязательно для адаптера).
- Docker Compose — воспроизводимый запуск за 5 минут.
- README с инструкцией быстрого старта.

## 11. Модель развёртывания и лицензирования

### Варианты деплоя MVP

| Вариант | Описание | Целевой клиент |
|---|---|---|
| **On-premise** | Docker Compose на сервере клиники | ДЦ с требованием хранить данные локально; IT-персонал есть |
| **VPS/ЦОД РК** | Docker Compose на арендованном VPS (Казтелеком, KDDI, Jusan Cloud) | ДЦ без своего сервера; данные остаются в РК |
| **SaaS multi-tenant** | Облако с изоляцией по organization_id | Задел; реализуется после MVP при масштабировании |

### Лицензионная модель (рекомендация)

- **MVP → пилот:** бесплатно или по себестоимости для 1–2 пилотных ДЦ.
- **Коммерческий запуск:** подписка SaaS (месяц / год) по числу модальностей или пользователей.
- **On-premise:** разовая лицензия + годовая поддержка.
- Альтернатива: открытый исходный код (open core) — базовая RIS бесплатно, расширения платно.
