## 1. Назначение и контекст

Радиологическая информационная система (RIS) автоматизирует рабочий процесс отделения
лучевой диагностики / диагностического центра: от регистрации пациента и направления —
до проведения исследования, описания врачом-рентгенологом и выдачи заключения.

**Классический поток:** регистрация → направление → запись/расписание → передача задания
на аппарат (DICOM MWL) → проведение исследования → приём снимков → описание по шаблону →
подпись и выдача заключения → статистика/метрики.

**Контекст MVP:** самостоятельная (standalone) система для частного диагностического
центра/клиники. Архитектура **integration-ready** — приём направлений изолирован за
интеграционным адаптером, чтобы позже добавить вход из МИС (HL7) без переписывания ядра.

### Ориентиры (best practices ведущих систем)
- **GE Centricity RIS** — веб-описание, структурированные шаблоны протоколов, метрики
  turnaround time, дашборды нагрузки.
- **Agfa Enterprise Imaging** — задача-ориентированный workflow: исследование как набор
  задач (контроль качества → описание → подпись).
- **Epic Radiant, Siemens RIS** — единая карточка пациента, клинические worklist'ы.
- **Orthanc / dcm4chee / ThaiRIS** — зрелые open-source DICOM-компоненты.

## 2. Границы MVP

### В составе MVP
- Регистратура, пациенты, направления, расписание/запись.
- DICOM Modality Worklist (MWL), приём снимков (C-STORE), просмотр снимков.
- Описание исследований по шаблонам, подпись (ЭЦП-ready), выдача заключений.
- 6 ролей с ролевым доступом (RBAC).
- Справочники РК: тарификатор услуг, МКБ-10, аппараты, шаблоны протоколов.
- Аудит, статистика и метрики (включая turnaround time).
- cito-оповещения направителю; второе мнение/рецензия.

### Вне MVP (отложено, но с архитектурным заделом)
- Личный кабинет пациента.
- Полноценный биллинг (тарификатор используется как справочник услуг с тарифом).
- Интеграция с МИС по HL7 (только контракт адаптера).
- Интеграция с нацсистемами РК; СМС-оповещения пациенту.
- Казахская локализация контента (i18n-задел есть, перевод позже).
- Квалифицированная ЭЦП через NCALayer / НУЦ РК (логика подписи готова, интеграция позже).

## 3. Архитектура

**Вариант A — модульный монолит + Orthanc как отдельный DICOM-узел.**

```
Веб-фронтенд (SPA) — рабочие места 6 ролей
        │  REST/JSON + WebSocket (обновление статусов)
RIS Backend (модульный монолит)
   Модули: Пациенты/регистратура · Направления · Расписание · Worklist ·
           Описания/подпись · Справочники · Пользователи/роли/аудит ·
           Статистика/метрики · Интеграционный адаптер
        │  SQL                         │  DICOMweb / REST
   БД (PostgreSQL)                Orthanc (MWL · C-STORE · вьюер · хранилище DICOM)
                                       │  DICOM
                              Модальности: КТ, МРТ, РГ, УЗИ, ММГ …
```

**Принципы:**
- **Разделение ответственности:** RIS ведёт рабочий процесс и данные; Orthanc хранит и
  показывает DICOM-снимки. RIS **не дублирует пиксельные данные**, ссылается на них по
  идентификаторам.
- **Связка по Accession Number / Study Instance UID** прошивает заказ → MWL → снимки →
  автосопоставление.
- **Интеграционный адаптер** изолирует внешний мир (сейчас — Orthanc; задел — HL7/МИС,
  нацсистемы РК).
- **Сквозной аудит** всех действий с медданными.
- **Чёткие границы модулей** — каждый модуль тестируется независимо; задел под выделение
  в сервисы при росте.

**Деплой-топология:** RIS backend, БД и Orthanc — отдельные контейнеры в общей сети.
Orthanc хранит DICOM на своём томе, RIS — свою SQL-БД; раздельные бэкапы.

## 4. Роли и рабочие места

| Роль | Основные функции |
|---|---|
| **Регистратор** | Поиск/создание пациента; создание направления; запись в расписание; отметка явки; печать талона/направления; рабочий список записей |
| **Рентгенолаборант** | Рабочий список аппарата; статусы «прибыл/начато/снимки получены»; контроль сопоставления студии с заказом; контроль качества (принять/переснять) |
| **Врач-рентгенолог** | Рабочий список «к описанию»; просмотр снимков (вьюер Orthanc); описание по шаблону + МКБ-10; подпись и выдача заключения |
| **Заведующий отделением** | Обзор нагрузки и статусов; переназначение описаний; второе мнение/рецензия; контроль просроченных (turnaround time) |
| **Врач-направитель** | Создание направлений своим пациентам; просмотр статуса и готовых заключений (только чтение, только свои пациенты) |
| **Администратор** | Пользователи и роли; справочники (услуги, аппараты, шаблоны, МКБ); аудит, статистика, системные настройки |

**Доступ:** RBAC, принцип наименьших привилегий. Врач-направитель видит только своих
пациентов. Доступ к снимкам — только через RIS-сессию (вьюер Orthanc не публичен).

## 5. Функциональные требования

### M1. Пациенты и регистратура
- **F1.1** Поиск пациента по ИИН / ФИО / дате рождения / телефону.
- **F1.2** Создание/редактирование карточки: ИИН, ФИО, пол, дата рождения, контакты,
  льготная категория (задел под тарификацию).
- **F1.3** Контроль дублей по ИИН (предупреждение при совпадении).
- **F1.4** История исследований пациента (направления и заключения).

### M2. Направления (orders)
- **F2.1** Создание направления: пациент → услуга (тарификатор) → модальность → область →
  врач-направитель → приоритет (плановый/срочный/cito) → предв. диагноз (МКБ-10) →
  клинические сведения.
- **F2.2** Генерация **Accession Number** и идентификатора заказа.
- **F2.3** Управление статусами направления (см. раздел 7).
- **F2.4** Отмена направления с указанием причины.
- **F2.5** Печать направления (форма с учётом НПА РК).

### M3. Расписание и запись
- **F3.1** Календарь по аппаратам (день/неделя), слоты по длительности услуги.
- **F3.2** Запись направления в свободный слот; перенос/отмена.
- **F3.3** Учёт рабочего времени аппарата и недоступности (ТО, поломка).
- **F3.4** Отметка явки пациента («прибыл»).
- **F3.5** Предупреждение о конфликтах/двойной записи.

### M4. Worklist и проведение исследования
- **F4.1** Публикация запланированных исследований в **DICOM MWL** (Orthanc): демография +
  Accession Number + Study Instance UID + AE Title аппарата.
- **F4.2** Рабочий список лаборанта по аппарату.
- **F4.3** Статусы: к выполнению → прибыл → начато → снимки получены.
- **F4.4** Автосопоставление пришедшей в Orthanc студии с заказом по Accession Number /
  Study Instance UID.
- **F4.5** Ручное связывание студии и заказа при расхождении (очередь «не сопоставлено»).
- **F4.6** Контроль качества: принять снимки / переснять (возврат в IN_PROGRESS),
  комментарий лаборанта.

### M5. Описание и заключение
- **F5.1** Рабочий список «к описанию» с фильтрами (модальность, приоритет, срок).
- **F5.2** Экран описания: данные направления + просмотр снимков (встроенный вьюер Orthanc).
- **F5.3** Шаблоны протоколов по модальности/области: структурированные поля + свободный текст.
- **F5.4** Заключение + код(ы) МКБ-10.
- **F5.5** Черновик → подпись (ЭЦП-ready: врач + дата/время + неизменяемость + хэш
  содержимого).
- **F5.6** Выдача заключения (PDF/печать); пометка критических находок → **cito-оповещение
  направителю**.
- **F5.7** Переназначение и **второе мнение/рецензия** (инициирует заведующий).

### M6. Справочники
- **F6.1** Тарификатор услуг (код, наименование, тариф, модальность, длительность).
- **F6.2** МКБ-10 (поиск по коду/наименованию).
- **F6.3** Аппараты/модальности (тип, AE Title, расписание, статус).
- **F6.4** Шаблоны протоколов: **механизм создания/редактирования** администратором
  (контент заполняется позже); привязка к услуге/модальности.
- **F6.5** Пользователи, роли, права.

### M7. Пользователи, доступ, аудит
- **F7.1** Аутентификация (логин/пароль; задел под 2FA/СМС).
- **F7.2** RBAC: 6 ролей, разграничение доступа к данным.
- **F7.3** Сквозной аудит-лог: кто, что, когда (создание/изменение/просмотр/подпись).
- **F7.4** Неизменяемость подписанных документов; версионирование описаний.

### M8. Статистика и метрики
- **F8.1** Дашборд: число исследований по периодам/модальностям/врачам/аппаратам.
- **F8.2** **Turnaround time** (ISSUED − NEW) и доля просроченных.
- **F8.3** Нагрузка на аппараты и врачей.
- **F8.4** Выгрузка отчётов (CSV/Excel).

### M9. Интеграционный адаптер
- **F9.1** DICOMweb/REST-обмен с Orthanc (MWL, уведомления о новых студиях, ссылки на вьюер).
- **F9.2** Задел: входной интерфейс направлений (эквивалент HL7 ORM) — контракт описан,
  реализация позже.
- **F9.3** Задел: исходящий интерфейс заключений (эквивалент HL7 ORU) и нацсистемы РК — позже.

## 6. Модель данных (ключевые сущности)

```
Patient (пациент)
  └─< Order (направление) ── ReferringPhysician
         │   ├ Service (услуга из тарификатора)
         │   ├ DiagnosisICD (предв. диагноз, МКБ-10)
         │   └ accession_number, priority, status
         │
         ├─ Appointment (запись) ── Device/Modality + слот времени
         ├─ Study (исследование) ── study_instance_uid, orthanc_id, выполнил Technologist
         └─ Report (заключение) ── protocol_template, structured_fields, free_text,
                conclusion + ICD-10, author, signed_at, signature (ЭЦП-ready),
                status (draft/signed/issued), версии, critical_finding, second_opinion→Report
```

**Справочники/системные:** `Service` (тарификатор), `DiagnosisICD` (МКБ-10),
`Device/Modality` (тип, AE Title, расписание), `ProtocolTemplate`, `User`/`Role` (RBAC),
`AuditLog` (entity, action, user, timestamp, before/after).

**Ключевые правила:**
- **Order — центральная сущность.** Order → Appointment → Study → основной Report.
- **Accession Number** генерируется на Order, протягивается в MWL → снимки → сопоставление.
- **Report неизменяем после подписи;** правка = новая версия с историей.
- **Second opinion** — отдельный связанный Report, оригинал не перезатирается.
- **i18n-задел:** наименования справочников хранятся с поддержкой нескольких языков;
  тарификатор и МКБ — **версионируются** (меняются приказами МЗ РК).

## 7. Жизненный цикл исследования (статусы)

```
NEW (новое) → SCHEDULED (записано; MWL уходит в Orthanc) → ARRIVED (прибыл)
   → IN_PROGRESS (идёт) → ACQUIRED (снимки получены; автосопоставление + QC)
   → TO_REPORT (к описанию) → REPORTING (черновик) → SIGNED (подписано, неизменяемо)
   → ISSUED (выдано; при critical_finding → cito-оповещение направителю)
```

**Боковые переходы:**
- **CANCELLED** — отмена с причиной из NEW/SCHEDULED/ARRIVED (до начала исследования).
- **Переназначение** (заведующий) на TO_REPORT/REPORTING — меняет исполнителя, статус не
  откатывается.
- **Второе мнение** — из SIGNED/ISSUED создаётся связанный Report, основной статус сохраняется.
- **Переснять** (QC) — ACQUIRED → IN_PROGRESS с сохранением попытки в истории.

| Переход | Кто меняет |
|---|---|
| NEW → SCHEDULED | Регистратор |
| → ARRIVED | Регистратор / Лаборант |
| → IN_PROGRESS, QC | Лаборант |
| → ACQUIRED | Система (событие Orthanc) |
| → TO_REPORT | Система / Лаборант (после QC) |
| REPORTING → SIGNED → ISSUED | Врач-рентгенолог |
| CANCELLED, переназначение, 2-е мнение | Регистратор / Заведующий |

**Метрики** строятся на временных метках статусов (turnaround time, время ожидания описания
= SIGNED − ACQUIRED и т.д.).

## 8. Интеграции

### 8.1 RIS ↔ Orthanc (DICOM) — в MVP
- **MWL (исходящий):** при SCHEDULED RIS публикует worklist-запись для Orthanc MWL-плагина
  (ФИО, ИИН/ID, пол, ДР, Accession Number, Study Instance UID, модальность, AE Title,
  дата/время, описание процедуры). Аппарат выполняет C-FIND и подтягивает демографию.
- **Приём снимков (C-STORE, входящий):** аппарат отправляет студию в Orthanc. RIS узнаёт
  о новой студии:
  - **Рекомендуется — webhook:** Lua/Python-плагин Orthanc на событие `OnStoredInstance`
    вызывает RIS (реакция в реальном времени, без лишней нагрузки).
  - **Запасной вариант — REST-поллинг** новых студий через Orthanc REST API (если плагины
    недоступны в окружении).
- **Автосопоставление:** RIS ищет Order по Accession Number / Study Instance UID из тегов
  студии → ACQUIRED; при несовпадении — очередь ручного связывания (F4.5).
- **Просмотр:** из экрана описания открывается встроенный вьюер Orthanc (Stone Web Viewer /
  Orthanc Explorer 2) по `study_instance_uid`. Свой вьюер не разрабатывается.
- **Протоколы:** DICOMweb (QIDO/WADO) + Orthanc REST API. AE Titles — в справочнике устройств.

### 8.2 ЭЦП — ЭЦП-ready в MVP
- В MVP: подпись = врач + дата/время + блокировка изменений + хэш содержимого заключения.
- Модуль подписи изолирован за интерфейсом `SignatureProvider` — задел под подключение
  **NCALayer / НУЦ РК** (квалифицированная ЭЦП) без изменения бизнес-логики.
- Подписанный документ неизменяем; PDF-заключение с реквизитами врача (позже — штамп ЭЦП).

### 8.3 Задел под МИС/HL7 — проектируется, не реализуется
- Входной интерфейс направлений (эквивалент HL7 ORM) — контракт готов; в MVP им пользуется
  только внутренний UI.
- Исходящий интерфейс заключений (HL7 ORU) и нацсистемы РК — позже.

## 9. Справочники РК (локализация)

- **9.1 Тарификатор услуг** — на основе приказа МЗ РК № ҚР ДСМ-170/2020 (ГОБМП/ОСМС).
  Поля: код, наименование, тариф, модальность, область, длительность слота. Импорт из
  Excel/CSV + ручное ведение администратором; версионирование. Тариф хранится, полный
  биллинг отложен.
- **9.2 МКБ-10** — полный справочник (поиск по коду/наименованию), разовый импорт, версия
  фиксируется. Используется в предв. диагнозе и заключении.
- **9.3 Аппараты/модальности** — наименование, тип (CT/MR/CR/DX/US/MG/RF…), AE Title,
  расписание, статус.
- **9.4 Шаблоны протоколов** — механизм создания администратором (контент позже), привязка
  к услуге/модальности.
- **9.5 Прочее** — врачи-направители, льготные категории (задел), языки (i18n).

**Соответствие НПА РК:** печатные формы направления и заключения проектируются под
требования НПА РК к медицинской документации (реквизиты организации, врача, пациента, даты,
ИИН, коды); электронные документы — с заделом под ЭЦП.

## 10. Нефункциональные требования

- **Безопасность:** RBAC (6 ролей, наименьшие привилегии); хэширование паролей; HTTPS/TLS
  между компонентами; доступ к снимкам только через RIS-сессию/токен; задел под 2FA.
- **Аудит и целостность:** неизменяемый аудит-лог; неизменяемость подписанных заключений
  (правки = версии); хэш содержимого при подписи.
- **Данные и приватность (РК):** обработка ПДн согласно законодательству РК; хранение
  данных на территории РК (учесть при выборе хостинга); раздельные бэкапы RIS-БД и
  Orthanc-хранилища; политика резервного копирования и восстановления.
- **Производительность/надёжность (ориентиры, уточнить с командой):** открытие рабочего
  списка < 2 с; реакция на новый снимок (webhook) — секунды; десятки одновременных
  пользователей; устойчивость к недоступности Orthanc (очередь сопоставления не теряется,
  статусы не зависают).
- **Локализация/удобство:** i18n-задел (RU+KZ) в UI и документах; форматы дат/чисел РК;
  адаптивный веб-интерфейс; роле-ориентированные экраны.
- **Сопровождаемость:** чёткие границы модулей; покрытие тестами ключевой логики (статусы,
  сопоставление, доступ); документация API интеграционного адаптера; контейнеризация.
