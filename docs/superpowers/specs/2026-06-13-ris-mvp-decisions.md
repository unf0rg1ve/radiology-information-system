# RIS MVP — Промежуточные решения и контекст

> Рабочий файл брейншторминга. Фиксирует принятые решения по ходу обсуждения.
> Дата: 2026-06-13. Итоговое ТЗ будет в `2026-06-13-ris-mvp-design.md`.

## Что строим
MVP радиологической информационной системы (RIS) для рынка **Казахстана (РК)**.
Цель текущего этапа — **техническое задание (ТЗ)** для команды разработки на основе
best practices и функционала ведущих RIS-систем.

## Ориентиры (топ-системы)
- Epic Radiant, Siemens RIS
- GE Centricity RIS — веб-описание, структурированные шаблоны протоколов, метрики turnaround time, дашборды
- Agfa Enterprise Imaging — задача-ориентированный workflow: исследование = набор задач (QC → описание → подпись)
- Облачные RIS/PACS: Simplirad, RadPortal
- Open-source: ThaiRIS, dcm4chee, Orthanc
- РФ-ориентиры: ЕРИС (Электрон/Москва), iRIS, ЛИНС RIS-X

## Принятые решения

| # | Тема | Решение |
|---|------|---------|
| 1 | Контекст развёртывания | **Standalone** (частный диагностический центр/клиника), но архитектура **integration-ready** — приём направлений через отдельный адаптер, чтобы позже добавить HL7/МИС без переписывания ядра |
| 2 | Глубина DICOM/PACS | **MWL + приём снимков (C-STORE) + просмотр**. PACS не пишем сами |
| 3 | DICOM-стек | **Orthanc** как DICOM-узел: MWL, C-STORE, хранилище, встроенный вьюер (Stone Web Viewer / Orthanc Explorer 2). Связь RIS↔Orthanc через DICOMweb/REST |
| 4 | Роли MVP | Регистратор, Рентгенолаборант, Врач-рентгенолог, Заведующий отделением, Врач-направитель, Администратор |
| 5 | Регион/регуляторика | **Казахстан**: тарификатор медуслуг (приказ МЗ РК № ҚР ДСМ-170/2020, ГОБМП/ОСМС), кодирование **МКБ-10**, требования НПА РК к электронным документам |
| 6 | Язык интерфейса/документов | **i18n-задел** в архитектуре (RU+KZ), контент MVP — русский; казахский перевод позже |
| 7 | Подпись заключений (ЭЦП) | **ЭЦП-ready**: логика подписания и статусы есть, в MVP — простая электронная подпись (врач+время); интеграция **NCALayer / НУЦ РК** — позже |
| 8 | Архитектура | **Вариант A — модульный монолит** + Orthanc как отдельный DICOM-узел. Веб-фронтенд (SPA). Чёткие границы модулей, задел под микросервисы/HL7 |
| 9 | Технологический стек | На усмотрение команды разработки; в ТЗ — обоснованные рекомендации с альтернативами |

## Вне границ MVP (отложено)
- Личный кабинет пациента
- Биллинг (полноценный); тарификатор используется как справочник услуг с кодами и тарифом
- Интеграция с МИС по HL7 (только задел через адаптер)
- Интеграция с нацсистемами РК
- Казахская локализация контента
- Полноценная квалифицированная ЭЦП через NCALayer

## Архитектура (раздел 1 — согласован)
- Веб-SPA → RIS Backend (модульный монолит) → БД (SQL) + Orthanc (DICOM)
- Модули backend: Пациенты/регистратура · Направления и расписание · Worklist · Описания/подпись ·
  Справочники · Пользователи/роли/аудит · Статистика/метрики · Интеграционный адаптер
- Связка заказа и снимков по Accession Number / Study Instance UID
- Сквозной аудит всех действий с медданными

## Дополнительные решения по ходу дизайна
- cito-оповещения направителю (F5.6) и второе мнение (F5.7) — **в MVP**
- Уведомление RIS о новых снимках — **webhook** через Orthanc-плагин (`OnStoredInstance`), рекомендация; REST-поллинг как запасной вариант (оба описаны в ТЗ)
- Шаблоны протоколов — **только механизм создания** администратором; контент заполняется позже
- Шаблоны/справочники — структура с поддержкой нескольких языков и версионированием (тарификатор, МКБ)
